# Knowledge-producer-service Specification

## Обзор

**Назначение**: автоматически собирать материалы по заданной теме, структурировать их под формат базы знаний, готовить черновики вопросов и сохранять **черновик** до проверки человеком. В документе системного дизайна этому соответствует **этап 0** («Наполнение базы знаний»).

Реализация — **линейный LLM-workflow**

**Технологии**: Python 3.11+, FastAPI (запуск задач, статус, health), LLM.

**Модель**: GPT-5.4 (сложная обработка текста), GPT-5.4-mini (обычный режим), GPT-5.4-nano (простые проверки и классификация).

Подробности **workflow**, инструментов и этапов — в `docs/system-design.md` (этап 0, таблица инструментов, раздел «Роли этапов»).


## Шаги (LLM-workflow)

1. Вход: тема и уровень сложности (и идентификатор задачи для оператора).
2. `web_search(query, num_results)` — поиск по ML-ресурсам (документация, статьи, учебники).
3. `fetch_url(url)` — загрузка страниц и извлечение текста.
4. **Ручная загрузка** (отдельно от вызова инструмента): текст, который оператор передал через UI или API, попадает в ту же очередь на разбор, что и содержимое страниц после шага 3.
5. **LLM** обрабатывает собранное:
   - ключевые концепции
   - вопросы (теория / практика)
   - структурирование теории (описание, формулы, примеры)
   - приведение к формату базы знаний
6. Проверка на дубликаты:
   - `search_knowledge_base(query, topic)` — близкие концепции (**Knowledge-base-crud-service**)
   - `search_questions(topics, difficulty, type)` и/или сопоставление по текстовой близости — близкие вопросы (**Questions-crud-service**)
7. `save_draft_material(title, content, topic, type)` — запись только в статусе черновика.
8. Оператор проверяет черновик и при необходимости публикует запись в рабочей базе через **admin UI** и CRUD; сервис сам не пишет в продакшен без подтверждения.

## Инструменты (параметры задаются JSON Schema)

| Инструмент | Назначение |
|------------|------------|
| `web_search(query, num_results)` | Поиск источников |
| `fetch_url(url)` | Извлечение текста |
| `search_knowledge_base(query, topic)` | Поиск похожих концепций в базе знаний |
| `search_questions(...)` | Поиск похожих формулировок вопросов |
| `save_draft_material(title, content, topic, type)` | Сохранение черновика до проверки оператором |

## Лимиты

- `web_search` / `fetch_url` на одну задачу — по конфигу (ориентир: до 5 / 10 вызовов на тему)
- Таймауты HTTP и LLM — по окружению
- Проверка человеком обязательна перед публикацией в базе знаний

## Retry / Fallback

| Сценарий | Retry | Fallback |
|----------|-------|----------|
| Превышен лимит запросов к API поиска | 3 попытки | Отложить задачу, уведомить оператора |
| Ошибка LLM или невалидный JSON | 1–2 попытки | Сохранить необработанный ответ модели и пометить задачу для ручной правки |
| Дубликат в базе знаний | — | В интерфейсе оператора: объединить с существующей записью или отклонить |

## Метрики

- Число обработанных тем и черновиков в день
- Доля обнаруженных дубликатов (до проверки оператором)
- Доля черновиков, одобренных оператором
- Время шага LLM и доля отказов

## Serving / Config

**Запуск**: Docker (python-slim), Kubernetes, Helm, ArgoCD.

**Конфигурация**: `ENVIRONMENT`, `LOG_LEVEL`, `LLM_MODEL`, лимиты `WEB_SEARCH` и `FETCH_URL`, базовые URL клиентов к Knowledge-base-crud-service и Questions-crud-service (вызовы по внутреннему API, без очереди Kafka).

**Секреты (Vault)**: `OPENAI_API_KEY`, `SEARCH_API_KEY` (или эквивалент провайдера поиска).

**Версии моделей**: GPT-5.4, GPT-5.4-mini, GPT-5.4-nano — выбор по качеству, задержке и стоимости (см. `docs/system-design.md`, подраздел о стоимости).

**Лимиты**: как в разделе «Лимиты».

## Observability / Evals

**Метрики (Prometheus)**: `web_search_calls_total`, `fetch_url_calls_total`, `llm_tokens_used`, `materials_draft_saved_total`, `human_approval_total`.

**Логи (Loki)**: JSON, уровень INFO; по политике приватности не логировать полный текст загруженных страниц.

**Трейсы (OpenTelemetry)**: внешние вызовы, LLM, обращения к CRUD, `save_draft_material`.

**Оценки качества**: насколько точно находятся дубликаты, насколько полезен черновик, как часто оператор отклоняет материал.
