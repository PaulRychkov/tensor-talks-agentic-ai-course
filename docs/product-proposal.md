# TensorTalks

## Задача

ML-специалисты (junior/middle) готовятся к техническим интервью месяцами, но часто не понимают, где именно проваливаются. Обратная связь после отказов формальная или отсутствует - нет конкретики, какие темы слабые, что подтянуть и в каком порядке. Компании со своей стороны тратят много ресурсов на субъективную оценку кандидатов и не могут стандартизировать процесс.

TensorTalks симулирует реальные технические ML-собеседования: проводит интервью по программе, анализирует каждый ответ, оценивает уровень и формирует персонализированный отчёт - ключевые ошибки, сильные стороны, рекомендации и материалы для подготовки.

Проект работает на стыке AI-EdTech и HR-Tech, объединяет AI-анализ, структурированную ML-базу вопросов и персональные рекомендации.

## Что уже есть

Система построена как набор микросервисов (Go + Python) с Kafka, Redis и PostgreSQL, есть CI/CD, Kubernetes, ArgoCD, Vault secrets; 14 сервисов: авторизация, BFF, управление сессиями, CRUD-сервисы для чатов/результатов/вопросов/базы знаний, сборщик интервью, AI-сервис.

Основной функционал:

- **Сборка программы интервью.** Interview Builder получает параметры (темы, уровень, тип), подтягивает вопросы и теорию из базы знаний и формирует программу. Это детерминированный процесс - без LLM.
- **LLM-workflow для процесса интервью.** AI Service ведёт интервью: загружает контекст, проверяет ответ на off-topic, оценивает ответ, принимает решение (уточнить / дать подсказку / перейти к следующему вопросу / завершить) и генерирует текст реплики. Каждый шаг вызывает LLM, но логика переходов довольно жесткая и задана приложением - это не автономный агент.
- **Оркестрация и состояние.** Dialogue Aggregator координирует процесс через Kafka-события, хранит состояние диалога в Redis (история, текущий вопрос, попытки), сохраняет сообщения и результаты в БД.
- **Результат.** После завершения интервью сохраняется оценка (score) и текстовая обратная связь. Пользователь видит их в интерфейсе.

Чего нет: подбор материалов под конкретные ошибки, структурированный отчёт с планом подготовки, evals и регулярная проверка качества. LLM задействован только в одном из четырёх этапов, и то как llm workflow, а не как агент.

## Что делаем в рамках курса

Цель PoC - превратить текущий LLM-workflow в агентную систему. Сейчас LLM работает только в процессе интервью и только как фиксированная цепочка. Наполнение базы знаний полностью ручное. Добавляем агентов с инструментами (включая внешние), памятью и автономным принятием решений.

**Наполнение базы знаний и вопросов.** Сейчас полностью вручную, никакой автоматизации. Добавляем агент, который с помощью внешних инструментов (`web_search`, `fetch_url`) ищет материалы по ML-темам, обрабатывает и структурирует их, формулирует вопросы и теорию, или возможно просто LLM-workflow. Черновики проверяет человек перед добавлением в базу (human-in-the-loop).

**Формирование интервью.** Сейчас детерминированная сборка с фильтрами. Добавляем агент-планировщик, который с помощью инструментов (поиск вопросов, проверка покрытия тем, валидация баланса/дублей, связи между темами) самостоятельно собирает программу c осмысленной траекторией и ростом сложности, а не случайной выборкой.

**Процесс интервью.** Сейчас LLM-workflow с жёсткой цепочкой шагов. Переделываем на агента-интервьюера с инструментами (оценка ответа, поиск теории в бд для подсказок, по необходимости `web_search` для проверки утверждений о свежих технологиях, суммаризация диалога). Агент сам выбирает следующее действие на основе состояния, а не по заданной цепочке.

**Отчёт и материалы.** Сейчас: число (score) + простая текстовая строка, подбора материалов нет. Добавляем: агент-аналитик, который анализирует оценки по всем вопросам, ищет материалы под пробелы (сначала во внутренней бд, если не хватает - через `web_search`), и формирует структурированный отчёт с планом подготовки.

Все агенты работают с общим состоянием сессии (state & memory через Redis/LangGraph). Инструменты оформлены как typed tools с чёткими схемами. Используем внешние инструменты (`web_search`, `fetch_url`). Добавляются guardrails (pre/post call checks), eval-набор и observability.

## Что покажу на демо

1. Пользователь выбирает уровень и тему
2. Агент-интервьюер проводит 5 вопросов: самостоятельно решает когда уточнить, подсказать (с опорой на бд), перейти дальше. Корректно обрабатывает "не знаю", "пропустим", уход от темы.
3. Агент-аналитик формирует структурированный отчёт: оценка, ключевые ошибки, подобранные материалы (из бд + внешние), план подготовки.

## Что не входит в PoC

- Полноценная production-инфраструктура для evals (достаточно набора тест-кейсов с ручным прогоном).
- Интеграции во внешние системы (HR/ATS, SSO).
- MCP-интеграции (Google Scholar, arxiv) - в перспективе.

## Метрики

Продуктовые: completion rate, повторное прохождение.

Агентские: tool call accuracy, качество подбора материалов (релевантность), качество отчёта (полнота, привязка к ошибкам).

Качество интервью: релевантность реплик, польза уточнений/подсказок, ложные off-topic.

Технические: задержка ответа (p95 < 8s), сохранность данных при сбоях LLM, стоимость (токены) на интервью.

## Edge cases

- "Не знаю / забыл" - агент даёт подсказку с опорой на базу знаний.
- Явный запрос подсказки - агент ищет релевантную теорию и формулирует подсказку.
- "Пропустим" - переходит дальше с записью в оценку.
- Уход от темы - агент определяет off-topic и возвращает к вопросу.
- Таймаут от LLM - fallback-ответ пользователю.
- Агент зацикливается на инструменте - лимиты на число вызовов.
- `web_search` возвращает нерелевантное - агент фильтрует и пробует другой запрос.

## Где агент, а где нет

Детерминированно: управление сессиями, хранение данных, маршрутизация Kafka-событий, валидация схем, лимиты и таймауты, rate limits.

Агент с инструментами: планирование интервью, ведение интервью, анализ результатов и подбор материалов, генерация отчёта.
