# TensorTalks

## Задача

ML-специалисты (junior/middle) готовятся к техническим интервью месяцами, но часто не понимают, где именно проваливаются. Обратная связь после отказов формальная или отсутствует - нет конкретики, какие темы слабые, что подтянуть и в каком порядке. Компании со своей стороны тратят много ресурсов на субъективную оценку кандидатов и не могут стандартизировать процесс.

TensorTalks симулирует реальные технические ML-собеседования: проводит интервью по программе, анализирует каждый ответ, оценивает уровень и формирует персонализированный отчёт - ключевые ошибки, сильные стороны, рекомендации и материалы для подготовки.

Проект работает на стыке AI-EdTech и HR-Tech, объединяет AI-анализ, структурированную ML-базу вопросов и персональные рекомендации.

## Что уже есть

Система построена как набор микросервисов (Go + Python) с Kafka, Redis и PostgreSQL, есть CI/CD, Kubernetes, ArgoCD, Vault secrets; 17 сервисов: авторизация, BFF, admin-BFF, управление сессиями, CRUD-сервисы для чатов/результатов/вопросов/базы знаний/пользователей, dialogue-aggregator, interview-builder, interviewer-agent, analyst-agent, knowledge-producer.

Основной функционал:

- **Сборка программы интервью.** Interview Builder получает параметры (темы, уровень, тип), подтягивает вопросы и теорию из базы знаний и формирует программу. Это детерминированный процесс - без LLM.
- **LLM-workflow для процесса интервью.** AI Service ведёт интервью: загружает контекст, проверяет ответ на off-topic, оценивает ответ, принимает решение (уточнить / дать подсказку / перейти к следующему вопросу / завершить) и генерирует текст реплики. Каждый шаг вызывает LLM, но логика переходов довольно жесткая и задана приложением - это не автономный агент.
- **Оркестрация и состояние.** Dialogue Aggregator координирует процесс через Kafka-события, хранит состояние диалога в Redis (история, текущий вопрос, попытки), сохраняет сообщения и результаты в БД.
- **Результат.** После завершения интервью сохраняется оценка (score) и текстовая обратная связь. Пользователь видит их в интерфейсе.

Чего не было: подбор материалов под конкретные ошибки, структурированный отчёт с планом подготовки, evals и регулярная проверка качества. LLM был задействован только в одном из четырёх этапов, и то как llm workflow, а не как агент.

## Что делаем в рамках курса

Цель PoC - превратить текущий LLM-workflow в агентную систему. Сейчас LLM работает только в процессе интервью и только как фиксированная цепочка. Наполнение базы знаний полностью ручное. Добавляем агентов с инструментами (включая внешние), памятью и автономным принятием решений.

**Наполнение базы знаний и вопросов.** Сейчас полностью вручную, никакой автоматизации. Добавляем агент, который с помощью внешних инструментов (`web_search`, `fetch_url`) ищет материалы по ML-темам, обрабатывает и структурирует их, формулирует вопросы и теорию, или возможно просто LLM-workflow. Черновики проверяет человек перед добавлением в базу (human-in-the-loop).

**Формирование интервью.** Сейчас детерминированная сборка с фильтрами. Добавляем агент-планировщик, который с помощью инструментов (поиск вопросов, проверка покрытия тем, валидация баланса/дублей, связи между темами) самостоятельно собирает программу c осмысленной траекторией и ростом сложности, а не случайной выборкой.

**Процесс интервью.** Сейчас LLM-workflow с жёсткой цепочкой шагов. Переделываем на агента-интервьюера с инструментами (оценка ответа, поиск теории в бд для подсказок, по необходимости `web_search` для проверки утверждений о свежих технологиях, суммаризация диалога). Агент сам выбирает следующее действие на основе состояния, а не по заданной цепочке.

**Отчёт и материалы.** Сейчас: число (score) + простая текстовая строка, подбора материалов нет. Добавляем: агент-аналитик, который анализирует оценки по всем вопросам, ищет материалы под пробелы (сначала во внутренней бд, если не хватает - через `web_search`), и формирует структурированный отчёт с планом подготовки.

Все агенты работают с общим состоянием сессии (state & memory через Redis/LangGraph). Инструменты оформлены как typed tools с чёткими JSON Schema. Используем внешние инструменты (`web_search`, `fetch_url`). Добавляются guardrails (pre/post call checks), eval-набор и observability.

**Structured output.** Все ответы LLM парсятся в Pydantic-модели: `AnswerEvaluation` (оценка ответа с полями completeness_score, accuracy_score, decision, missing_points, strong_points, decision_confidence, reasoning), `AnalystReport` (отчёт с summary, score, errors_by_topic, strengths, preparation_plan, materials), `ProgramMeta` (метаданные программы: validation_passed, coverage, fallback_reason), `OffTopicClassification`, `PIICheckResult`. Валидация через `model_validate_json` с fallback при невалидном JSON.

**Episodic memory.** При `use_previous_results=true` планировщик загружает историю сессий пользователя через `get_user_history(user_id, topics)` — слабые/сильные темы, scores. Интервьюер загружает `previous_topic_scores` из results-crud-service для персонализации: учитывает прогресс пользователя при выборе стратегии. Аналитик использует `ProgressDelta` для сравнения динамики между сессиями.

**Confidence scores.** `AnswerEvaluation` содержит поле `decision_confidence` (0.0–1.0) — уверенность агента в принятом решении. При высокой уверенности (≥ 0.7) и score в пограничной зоне (0.75–0.85) агент переходит к следующему вопросу; при низкой — выбирает безопасное действие (подсказка вместо перехода). При confidence < 0.5 возможна переоценка ответа (self-reflection). Метрики confidence записываются в Prometheus для мониторинга калибровки.

## Что покажу на демо

1. Пользователь регистрируется (автогенерация логина, recovery key — без email и персональных данных)
2. Пользователь выбирает уровень, тему и режим (интервью / тренировка / изучение темы)
3. Агент-планировщик формирует программу: подбирает вопросы из базы, проверяет покрытие тем, валидирует баланс и прогрессию сложности. Для study-режима генерирует иерархию: subtopic → теоретические блоки → контрольные вопросы
4. Агент-интервьюер проводит интервью: самостоятельно решает когда уточнить, подсказать (с опорой на базу знаний), перейти дальше. Оценивает уверенность в решениях (confidence scores) — при низкой уверенности выбирает безопасное действие. Корректно обрабатывает "не знаю", "пропустим", уход от темы
5. Агент-аналитик формирует структурированный отчёт (Pydantic-модель AnalystReport): оценка, ошибки по темам с пояснениями, сильные стороны, подобранные материалы (из базы знаний + внешние через arxiv), план подготовки. Создаёт пресеты для тренировок и изучения слабых тем
6. Пользователь изучает рекомендованную тему (study session), после чего разблокируется тренировка по ней

## Что не входит в PoC

- Полноценная production-инфраструктура для evals (достаточно набора тест-кейсов с ручным прогоном).
- Интеграции во внешние системы (HR/ATS, SSO).
- MCP-интеграции (Google Scholar, arxiv) - в перспективе.

## Метрики

Продуктовые: completion rate, повторное прохождение, hint usage rate, training unlock rate (доля пользователей, разблокировавших тренировку после study).

Агентские: tool call accuracy, качество подбора материалов (релевантность), качество отчёта (полнота, привязка к ошибкам), распределение confidence scores, доля решений с низкой уверенностью (< 0.5).

Качество интервью: релевантность реплик, польза уточнений/подсказок, ложные off-topic, калибровка score (совпадение с ручной оценкой).

Технические: задержка ответа (p95 < 8s), сохранность данных при сбоях LLM, стоимость (токены) на интервью. Prometheus-метрики: `agent_llm_call_duration_seconds`, `agent_decision_confidence`, `guardrail_triggered_total`.

## Edge cases

- "Не знаю / забыл" — агент даёт подсказку с опорой на базу знаний.
- Явный запрос подсказки — агент ищет релевантную теорию и формулирует подсказку.
- "Пропустим" — переходит дальше с записью в оценку (action=skip).
- Уход от темы — LLM-классификация off-topic, агент возвращает к вопросу.
- Персональные данные в сообщении — PII-фильтрация (regex + LLM), сообщение блокируется, пользователю объяснение.
- Низкая уверенность в оценке (confidence < 0.5) — переоценка ответа (self-reflection), при повторной низкой уверенности — безопасное действие.
- Таймаут от LLM — retry с exponential backoff (max 3), затем fallback-ответ пользователю.
- Агент зацикливается — лимит итераций (6 для интервьюера, 10 для аналитика, 8 для планировщика).
- `web_search` возвращает нерелевантное — агент фильтрует, пробует другой запрос; URL только из trusted domains.
- Агент-планировщик не может собрать программу — fallback на детерминированный pipeline.
- Агент-аналитик не может сгенерировать отчёт — fallback на линейный pipeline.

## Доработки не связанные с AI

**Регистрация и приватность (152-ФЗ).** Автогенерация логина (`{Прилагательное}{Существительное}{Число}`, например BrightNeural42) — без email и персональных данных. Recovery key для восстановления аккаунта. PII-фильтрация перед LLM: regex (email, телефон, ИНН, СНИЛС, паспорт, самоидентификация, компании из blacklist) + LLM-классификация неявных данных. Конфиденциальные данные не сохраняются в БД и не логируются.

**Три режима сессий.** Помимо интервью добавлены тренировка и изучение темы (study session). Study — иерархическая программа (subtopic → теоретические блоки с theory 500–1000 символов → контрольные вопросы), после прохождения разблокируется тренировка. Тренировка — приоритет практическим вопросам (70%), может запускаться из пресета после интервью.

**Admin-панель.** Отдельный frontend (admin-frontend) с admin-bff-service: управление базой знаний и вопросами (CRUD), очередь черновиков от knowledge-producer (HITL — просмотр, approve/reject, публикация), метрики (технические, AI, продуктовые) по кнопке.

**Дашборд пользователя.** Карточки с рекомендациями после интервью: кнопки «Изучить тему» и «Тренировка» на основе пресетов аналитика. Отображение прогресса по темам и разблокировка тренировок.

**Результаты.** Структурированный отчёт с секциями (Summary, Errors by topic, Strengths, Preparation plan, Materials), scores по темам, пресеты для follow-up сессий. Для study — study_plan и theory_reviewed.

**Cloudflared tunneling.** Доступ к платформе через https://www.tensor-talks.ru/ без статического IP.

## Где агент, а где нет

Детерминированно: управление сессиями, хранение данных, маршрутизация Kafka-событий, валидация схем, лимиты и таймауты, rate limits, PII-фильтрация (regex-уровень), оркестрация (dialogue-aggregator).

Агент с инструментами: планирование интервью (агент-планировщик), ведение интервью (агент-интервьюер), анализ результатов и подбор материалов, генерация отчёта (агент-аналитик).

LLM-workflow (без агентного графа): наполнение базы знаний (knowledge-producer), PII-классификация (Level 2), off-topic detection.
